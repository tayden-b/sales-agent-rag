# Product Requirements Document: Sales Call Intelligence System

## 1. Product Overview

**Product Name:** Sales Call Intelligence System
**Version:** 1.0
**Author:** Tayden Barretto
**Date:** 2026-02-06

### Vision
An AI-powered system that transforms raw sales call transcripts into actionable intelligence — delivered as a structured email — using multi-agent orchestration and retrieval-augmented generation (RAG). The system learns from every processed call, building an organizational memory that improves analysis over time.

### Problem Statement
Sales engineers and account executives spend significant time after customer calls manually:
- Summarizing key discussion points
- Looking up documentation to address technical concerns
- Identifying and prioritizing follow-up actions
- Comparing the deal to similar past situations
- Drafting internal communications about deal health

This manual process is slow, inconsistent, and loses institutional knowledge when it stays in individual notes.

### Solution
An automated pipeline where a user uploads a call transcript and receives a comprehensive, well-formatted email containing an executive summary, technical concerns matched to relevant documentation, prioritized action items, resources to share with the customer, and internal deal health assessment — all generated by coordinated AI agents with access to documentation and historical call data.

## 3. Feature Specifications

### 3.1 Transcript Ingestion
- **Input Methods:**
  - Upload a `.txt` file containing the call transcript
  - Paste raw transcript text via CLI prompt
- **Validation:**
  - Reject empty or extremely short transcripts (< 50 words)
  - Warn on very long transcripts (> 50,000 words) — still process them
  - Extract account name and date from transcript if present, or prompt user
- **Output:** Cleaned transcript text ready for agent processing

### 3.2 Multi-Agent Analysis Pipeline
Five specialized agents process the transcript sequentially:

#### Agent 1: Transcript Analyzer
- **Input:** Raw transcript text
- **Output (JSON):**
  - `account_name`: string
  - `call_date`: string
  - `account_stage`: enum (Discovery, Evaluation, POC, Negotiation, Renewal, Expansion)
  - `products_discussed`: list of strings (Vault, Terraform, Consul, etc.)
  - `technical_concerns`: list of objects `{concern, context_quote, severity}`
  - `pain_points`: list of strings
  - `competitors_mentioned`: list of strings
  - `action_items`: list of objects `{action, owner, priority}`
  - `sentiment`: enum (Positive, Neutral, Cautious, Negative)
  - `key_quotes`: list of strings (notable customer statements)

#### Agent 2: Documentation Researcher
- **Input:** `technical_concerns` and `products_discussed` from Agent 1
- **Tool:** RAG search over `hashicorp-docs` collection
- **Output (JSON):**
  - For each concern: `{concern, relevant_docs: [{title, url, excerpt, relevance_score}], recommendation}`

#### Agent 3: Historical Context Analyst
- **Input:** `account_stage`, `products_discussed`, `technical_concerns` from Agent 1
- **Tool:** RAG search over `call-transcripts` collection
- **Output (JSON):**
  - `similar_calls`: list of `{account, date, similarity_reason, outcome}`
  - `patterns`: list of strings (observed patterns across similar situations)
  - `suggestions`: list of strings (what worked before in similar deals)

#### Agent 4: Email Composer
- **Input:** Outputs from Agents 1, 2, and 3
- **Output:** Formatted HTML email following the template defined in Section 3.3

#### Agent 5: Knowledge Base Updater
- **Input:** Raw transcript + structured analysis from Agent 1
- **Tool:** Vector database write operations
- **Output:** Confirmation that transcript is indexed with metadata

### 3.3 Email Output Format
```
Subject: Call Summary: [Account Name] - [Date]

## Executive Summary
- [3-4 key takeaways as bullet points]

## Technical Concerns & Documentation

### Concern 1: [Brief description]
**Context**: [What the customer said/asked]
- **Documentation**: [Link to relevant HashiCorp doc]
- **Recommendation**: [How to address this concern]

[Repeat for each concern]

## Recommended Next Steps
1. [Highest priority action] - Owner: [Who]
2. [Next priority action] - Owner: [Who]
3. [Additional actions...]

## Resources to Share with Customer
- [Link 1]: [Brief description]
- [Link 2]: [Brief description]

## Internal Notes
- **Deal Health**: [Healthy / At-Risk / Needs Attention]
- **Key Risks**: [Blockers or concerns]
- **Historical Context**: [Similar deals and patterns]
- **Strategic Notes**: [Observations and recommendations]

---
Generated by Call Intelligence AI
```

### 3.4 RAG Knowledge Base
- **Collection: `hashicorp-docs`**
  - Pre-populated with HashiCorp Vault and Terraform documentation
  - Chunked with context preservation (no mid-sentence splits)
  - Tagged with: product, topic, doc_type (guide, reference, tutorial)
  - Searchable by semantic similarity + metadata filters

- **Collection: `call-transcripts`**
  - Populated automatically as transcripts are processed
  - Tagged with: account_name, call_date, account_stage, products_discussed, sentiment
  - Searchable by semantic similarity + metadata filters
  - Grows with each processed call

### 3.5 Email Delivery
- **Provider:** SendGrid API (primary), SMTP fallback
- **Features:**
  - HTML formatted email with plain-text fallback
  - Configurable recipient(s) via environment variable
  - Email preview mode for development (print to console instead of sending)
  - Sent email logging for debugging

## 4. Technical Requirements

| Requirement | Specification |
|-------------|---------------|
| Language | Python 3.11+ |
| Agent Framework | CrewAI |
| LLM Provider | OpenAI API (GPT-4o-mini for agents, cost-efficient) |
| Vector Database | ChromaDB (local, free, simple) |
| Embeddings | OpenAI `text-embedding-3-small` |
| Email Service | SendGrid API |
| Configuration | `.env` file with `.env.example` template |
| Error Handling | Graceful failures with clear log messages |
| Logging | Python `logging` module, structured output |

### Non-Functional Requirements
- Processing time: Target < 3 minutes for a typical transcript (2000-5000 words)
- Reliability: System should not crash on malformed input
- Observability: Each agent logs start/finish and key outputs
- Portability: Runs on any machine with Python 3.11+ and API keys

## 6. Out of Scope (v1.0)

The following are explicitly **not** included in v1.0 but are noted as potential future enhancements:

- **Web UI / Dashboard** — v1.0 is CLI-only
- **Real-time call transcription** — System processes pre-existing transcripts only
- **CRM Integration** (Salesforce, HubSpot) — No automatic deal updates
- **Competitive intelligence database** — Competitor mentions are noted but not deeply analyzed
- **Multi-language support** — English transcripts only
- **User authentication / multi-tenancy** — Single-user system
- **Call recording ingestion** — Audio-to-text is out of scope; input is text transcripts
- **Automated scheduling** — No cron jobs or recurring processing
- **Custom email templates** — Fixed format in v1.0

## 7. Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| LLM produces inaccurate analysis | Medium | Use structured output (JSON) with validation; include confidence scores |
| RAG returns irrelevant documents | Medium | Set relevance threshold; test with real transcripts; tune chunk size |
| API rate limits (OpenAI, SendGrid) | Low | Add retry logic with exponential backoff |
| ChromaDB data loss (local storage) | Low | Document backup procedures; data is regenerable |
| Long transcripts exceed token limits | Medium | Implement transcript chunking for very long calls |
| Email marked as spam | Low | Use SendGrid with proper domain verification |
